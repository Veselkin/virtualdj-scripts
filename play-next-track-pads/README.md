# Play Next Track (пады → плейлисты)

## Задача

Есть MIDI-контроллер с несколькими пэдами. Нужно, чтобы при нажатии на пэд:
1. Воспроизводился **следующий трек** из конкретного плейлиста/папки (у каждого пэда — свой плейлист).
2. При достижении конца плейлиста — воспроизведение зацикливалось на первый трек.
3. Повторные нажатия работали стабильно, без пропусков и повторов одного и того же трека.

## Файлы

- [`mapping.xml`](./mapping.xml) — готовый маппинг для 5 падов в формате VirtualDJ (страница `custom`).

## Установка маппинга

Файл `mapping.xml` — это фрагмент страницы `custom` из внутреннего формата маппингов VirtualDJ (Pads Editor → вкладка **custom**). Варианта применить два:

1. **Через интерфейс** — открыть Pads Editor, выбрать страницу `custom`, и для каждого пада вручную вставить нужную строку скрипта в поле Action (см. содержимое `mapping.xml`, между тегами `<padN>...</padN>` — это и есть содержимое поля Action).
2. **Напрямую в файл маппинга контроллера** — найти XML-файл своего маппинга контроллера в папке VirtualDJ (`Documents\VirtualDJ\Mappers\`), найти в нём страницу `<page name="custom">` и вставить/заменить блок падов на содержимое `mapping.xml`. Требует осторожности — перед правкой сделать бэкап оригинального файла маппинга.

## Итоговый рабочий скрипт (на примере одного пэда)

Назначается на пэд через **Config → Controllers → Mapping → Learn**, вставляется в поле **Action** (Push Action):

```
deck 1 stop & browser_gotofolder "1 Select" & browser_window 'songs' & wait 100ms & browser_scroll +1 & deck 1 load & deck 1 play
```

Для остальных пэдов — та же конструкция, меняется только название папки (см. `mapping.txt`).

> Название папки в кавычках (`"1 Select"`) должно точно совпадать с именем плейлиста/папки в сайдбаре VirtualDJ, включая регистр и пробелы.

## Как это работает

1. `deck 1 stop` — останавливает деку. Это важно: VirtualDJ не всегда корректно подменяет трек командой `load`, если дека находится в состоянии play.
2. `browser_gotofolder "1 Select"` — переключает браузер файлов на нужную папку/плейлист.
3. `browser_window 'songs'` — принудительно фокусирует именно список треков (а не сайдбар/Automix/Sidelist, которые могут случайно оказаться в фокусе).
4. `wait 100ms` — небольшая пауза, чтобы интерфейс VirtualDJ успел перерисоваться после переключения папки, прежде чем скрипт продолжит выполнение.
5. `browser_scroll +1` — сдвигает выделение на следующий трек в списке.
6. `deck 1 load` — загружает выделенный (уже сдвинутый) трек в деку.
7. `deck 1 play` — запускает воспроизведение.

Порядок команд принципиален: `browser_scroll +1` стоит **до** `load`, а не после. Это сделано намеренно, чтобы весь цикл "сдвинуть → загрузить → сыграть" укладывался в одно нажатие пэда целиком и не зависел от завершения предыдущего вызова в фоне (если нажимать пэды быстро подряд).

## Особенности и известные ограничения

- **Первое нажатие после запуска** сразу сдвигает выделение на трек вперёд от текущего (то есть если выделен трек №1, сыграет трек №2). Это следствие того, что `scroll +1` идёт первым шагом. Решается вручную: перед первым нажатием пэда в начале сессии просто выделить трек, предшествующий тому, с которого нужно начать.
- **Зацикливание на первый трек по достижении конца плейлиста** реализуется штатным поведением `browser_scroll` — при попытке сдвинуться за пределы списка выделение возвращается/остаётся на границе списка в зависимости от версии VDJ. Если автоматического возврата в начало не происходит, потребуется доработка (см. TODO).
- Скрипт использует `deck 1` — если нужно воспроизводить пэды на разных деках одновременно (не последовательно на одной), номер деки нужно менять под конкретный пэд.

## Грабли, на которые наступили по пути (и как их обошли)

| Проблема | Причина | Решение |
|---|---|---|
| `browser_gotofolder 1` вёл не туда, включался трек из последнего открытого вручную списка | Числовые индексы `browser_gotofolder` ссылаются на зарегистрированные Favorite Folders, а не на любую папку в сайдбаре | Обращаться к папке по имени в кавычках: `browser_gotofolder "1 Select"` |
| После нескольких нажатий навигация "прыгала" по Automix/Sidelist вместо нужного плейлиста | VirtualDJ хранит общий фокус браузера, и сторонние действия (в т.ч. известный баг — автопереключение sideview на вкладку Automix ближе к концу трека) сбивают его | Явно фиксировать фокус командой `browser_window 'songs'` перед каждым `scroll` |
| Трек, загруженный в деку, был "на 1 больше" выделенного | `browser_scroll +1` стоял до `load`, из-за чего каждое нажатие сразу же уезжало на трек вперёд | Учтено как особенность поведения (см. выше) |
| При проигрывании трека повторное нажатие не переключало на следующий | В VirtualDJ есть защита от случайной подмены трека на играющей деке (влияет на поведение `load`) | Добавлен `deck 1 stop` перед `load` |
| Добавление `unload deck 1` перед `load` не помогло, а местами добавляло нестабильности | `load` сам по себе выгружает старый трек и грузит новый — отдельный `unload` избыточен и добавляет лишнюю операцию перерисовки интерфейса | `unload deck 1` убран из финального скрипта |
| Попытка передать номер шага через переменную (`get_var`, `get_text`, `param_cast`) в `browser_scroll` не сработала — воспроизводился только первый трек | `browser_scroll`, судя по всему, принимает только литеральные значения (`+1`, `-1`, `'top'`, `'bottom'`), а не вычисляемые параметры | Отказались от подхода со счётчиком, оставили пошаговый `scroll +1` |
| Нестабильное поведение ("то нормально, то повторяет один и тот же трек") при быстрых повторных нажатиях | Гонка состояний: новый вызов скрипта стартовал раньше, чем у предыдущего успевал доработать хвост (`wait` + `scroll` в конце) | `browser_scroll +1` перенесён в начало скрипта, до `load`/`play`, чтобы каждый вызов был полностью самодостаточным |
